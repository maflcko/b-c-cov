      69                 :            :  * The wallet holds one (for pre HD-split wallets) or several keypools. These

      71                 :            :  * change.
      72                 :            :  *
      73                 :            :  * The Bitcoin Core wallet was originally a collection of unrelated private
      74                 :            :  * keys with their associated addresses. If a non-HD wallet generated a

      77                 :            :  * lost definitively.
      78                 :            :  *
      79                 :            :  * The keypool was implemented to avoid this scenario (commit: 10384941). The
      80                 :            :  * wallet would generate a set of keys (100 by default). When a new public key
      81                 :            :  * was required, either to give out as an address or to use in a change output,
      82                 :            :  * it would be drawn from the keypool. The keypool would then be topped up to
      83                 :            :  * maintain 100 keys. This ensured that as long as the wallet hadn't used more
      84                 :            :  * than 100 keys since the previous backup, all funds would be safe, since a
      85                 :            :  * restored wallet would be able to scan for all owned addresses.
      86                 :            :  *
      87                 :            :  * A keypool also allowed encrypted wallets to give out addresses without
      88                 :            :  * having to be decrypted to generate a new private key.
      89                 :            :  *
      90                 :            :  * With the introduction of HD wallets (commit: f1902510), the keypool
      91                 :            :  * essentially became an address look-ahead pool. Restoring old backups can no
      92                 :            :  * longer definitively lose funds as long as the addresses used were from the
      93                 :            :  * wallet's HD seed (since all private keys can be rederived from the seed).
      94                 :            :  * However, if many addresses were used since the backup, then the wallet may
      95                 :            :  * not know how far ahead in the HD chain to look for its addresses. The
      96                 :            :  * keypool is used to implement a 'gap limit'. The keypool maintains a set of
      97                 :            :  * keys (by default 1000) ahead of the last used key and scans for the

      99                 :            :  * involving the wallet's addresses, or of re-using the same address.

     103                 :            :  * old backup. For this reason, it is not recommended to decrease keypool size
     104                 :            :  * lower than default value.
